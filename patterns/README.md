# Patterns

> Proven architecture patterns, templates, and reference implementations

## Contents

Each pattern document follows the structure:

1. **Context** -- When does this pattern apply?
2. **Problem** -- What problem does it solve?
3. **Solution** -- How to implement it
4. **Consequences** -- Trade-offs and considerations
5. **Example** -- At least one concrete example

## Naming Convention

`{pattern-name}.md` -- lowercase, hyphenated

## Contributing

1. Validate the pattern in a real project
2. Document with the standard structure above
3. Include concrete examples
4. Submit via PR
